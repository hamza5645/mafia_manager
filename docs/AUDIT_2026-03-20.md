# Mafia Manager Audit

Date: 2026-03-20

Scope: solo flow, multiplayer flow, Supabase RPC/schema, and test/build configuration.

Verification performed:
- Xcode workspace build succeeded.
- `xcodebuild ... test` failed with `Scheme mafia_manager is not currently configured for the test action.`
- Xcode snippet verification confirmed two solo logic bugs:
  - a 2 Mafia vs 2 Non-Mafia state after night does not end the game;
  - a tied vote advances directly to night without incrementing `dayIndex` or recording `dayHistory`.
- Live Supabase inspection confirmed backend drift and data anomalies:
  - `resolve_night_atomic()` is still deployed with camelCase key lookups;
  - live `night_history` payloads use snake_case keys;
  - all 16 live sessions with `day_index >= 2` currently store only 1 night-history record.

## Confirmed Defects

### MM-01 [Critical] `resolve_night_atomic()` is reading the wrong JSON keys and can wipe prior night history

Status: Fixed on 2026-03-21

Verification update:
- Deployed `resolve_night_atomic()` now reads `night_index` first and tolerates legacy `nightIndex` only as a fallback.
- Deployed `resolve_night_atomic()` now rejects malformed payloads with SQLSTATE `22023` instead of silently nulling `v_night_index`.
- Live verification still shows `night_history` payloads use snake_case keys; the backend drift described below has been corrected going forward.
- This is a fix-forward change only; previously collapsed `night_history` arrays were not backfilled by MM-01.

Confidence: High

Files:
- `Core/Models/Multiplayer/GameSession.swift`
- `Core/Services/Multiplayer/SessionService.swift`
- `supabase/multiplayer_schema.sql`

Evidence:
- `NightActionRecord` encodes snake_case keys such as `night_index` and `is_resolved`.
- `SessionService.resolveNightAtomic()` sends that encoded JSON record to Supabase.
- `resolve_night_atomic()` reads `p_night_record->>'nightIndex'` and filters existing history with `elem->>'nightIndex'`.
- Live database check:
  - 93 sessions contain `night_index` keys in `night_history`;
  - 0 sessions contain `nightIndex`;
  - all 16 sessions with `day_index >= 2` have `night_history` length `<= 1`.

Impact:
- `v_night_index` becomes `NULL`.
- The SQL filter does not match existing entries correctly.
- `night_history` can collapse to just the latest record on each atomic resolution.
- `removal_note = 'Eliminated at night ' || v_night_index` also becomes null-ish behavior because `v_night_index` is null.

Key references:
- `Core/Models/Multiplayer/GameSession.swift` encodes `night_index`
- `Core/Services/Multiplayer/SessionService.swift:356-454`
- `supabase/multiplayer_schema.sql:484-539`

Why players would feel this:
- later rounds lose accurate history;
- bot targeting and summaries can drift because prior nights silently disappear.

### MM-02 [Critical] Multiplayer doctor saves can be lost when doctors split their protections

Status: Fixed in app on 2026-03-21

Verification update:
- `NightActionRecord` now persists `target_was_saved`, so phase 1 stores the authoritative save verdict instead of relying on a summarized protection target.
- `recordNightActions()` now records that verdict and biases `doctorProtectedId` toward the mafia target when any doctor successfully saved them.
- `resolveNightOutcome()` now resolves from the stored verdict, with a compatibility fallback for unresolved legacy records that still lack `target_was_saved`.
- `MultiplayerNightView` no longer recomputes save state from `mafiaTargetId == doctorProtectedId`; the store remains the source of truth.
- Multiplayer summaries now read `target_was_saved` when available so saved nights are reported consistently.
- App build verification succeeded after the fix.
- Live room verification reproduced the split-doctor setup on 2026-03-21; final end-to-end morning-screen confirmation was partially blocked by a separate host night-advance/readiness issue.

Confidence: High

Files:
- `Core/Store/MultiplayerGameStore.swift`
- `Features/Multiplayer/MultiplayerNightView.swift`

Evidence:
- `recordNightActions()` correctly computes `targetWasSaved` as “did any doctor protect the mafia target?”
- The same method then stores only one `doctorProtectedId`, preferring the majority/tiebreak target.
- `MultiplayerNightView.determineIfTargetWasSaved()` later recomputes save status from `mafiaTargetId == doctorProtectedId`.

Impact:
- If one doctor protects the mafia target and another protects someone else, `targetWasSaved` can be `true` during recording but `false` during resolution.
- The victim dies even though at least one doctor protected them.

Key references:
- `Core/Store/MultiplayerGameStore.swift:1988-1995`
- `Features/Multiplayer/MultiplayerNightView.swift:240-247`
- `Features/Multiplayer/MultiplayerNightView.swift:671-679`

Why players would feel this:
- “some people die when they shouldn’t” exactly matches this failure mode.

### MM-03 [High] Multiplayer role assignment events can be dropped during lobby with no recovery path for clients

Status: Fixed in app on 2026-03-21

Verification update:
- `handlePlayerUpdate()` still ignores role-bearing `session_players` events while the local phase is `lobby`, preserving the stale-event guard for rematch/reset safety.
- `handleSessionUpdate()` now detects lobby-boundary transitions and forces a `refreshPlayers()` snapshot recovery when clients move from `lobby` into gameplay.
- The same boundary handling now refreshes players when returning from gameplay to `lobby`, clearing stale role, number, and mafia-teammate state if reset events were missed.
- Targeted regression coverage was added for dropped role-assignment recovery, lobby reset clearing, and the safety guard that still rejects role-bearing lobby events.
- App build verification succeeded after the fix; CLI `xcodebuild ... test` remains blocked because scheme `mafia_manager` is not configured for the test action.

Confidence: High

Files:
- `Core/Store/MultiplayerGameStore.swift`

Evidence:
- `handlePlayerUpdate()` discards any player update that contains a role while `currentPhase == "lobby"`.
- `handleSessionUpdate()` does not force a `refreshPlayers()` for non-host clients when phase changes out of lobby.

Impact:
- If role assignment updates arrive before the session phase update, non-host clients can miss their own role permanently for that round.
- Mafia teammates can remain empty.
- Players can land in role reveal or night with `myRole == nil`.

Key references:
- `Core/Store/MultiplayerGameStore.swift:600-635`
- `Core/Store/MultiplayerGameStore.swift:654-663`

Why players would feel this:
- some players cannot act at night;
- mafia players may not see teammates;
- phase flow appears stuck or inconsistent across devices.

### MM-14 [Medium] Multiplayer game-over winner can be temporarily or permanently blank after a night win

Confidence: High

Files:
- `Features/Multiplayer/MultiplayerGameOverView.swift`
- `Core/Services/Multiplayer/SessionService.swift`

Evidence:
- `MultiplayerGameOverView` reads the winner only from `currentSession?.winner`.
- `resolveNightAtomic()` updates phase/phaseData first and then does a second write for `is_game_over` and `winner`.
- The phase data already contains `.gameOver(winner: ...)`, but the view ignores it.

Impact:
- After a night win, clients can reach the game-over screen and still render `winner == nil`.
- If the follow-up winner update is delayed or missed, the UI shows “GAME ENDED / No Winner Determined” even when the phase data already knows the winner.

Key references:
- `Features/Multiplayer/MultiplayerGameOverView.swift:13-15`
- `Core/Services/Multiplayer/SessionService.swift:420-441`

### MM-15 [Medium] Multiplayer night death reveal can mislabel dead special roles as Citizen

Confidence: High

Files:
- `Features/Multiplayer/MultiplayerLobbyView.swift`

Evidence:
- `MultiplayerDeathRevealView.roleFor(player:)` falls back to inferring role from `mafiaPlayerNumbers`, `doctorPlayerNumbers`, and `inspectorPlayerNumbers`.
- Those arrays only contain players who actually submitted actions that night.
- A killed doctor or inspector who failed to submit will not appear in those arrays and is therefore classified as `.citizen`.

Impact:
- Death reveal can show the wrong role to non-host players.
- This is most likely in AFK / reconnect / missed-action scenarios, which are already the unstable edge cases players complain about.

Key references:
- `Features/Multiplayer/MultiplayerLobbyView.swift:659-680`
- `Features/Multiplayer/MultiplayerLobbyView.swift:727-741`

## Recommended Fix Order

1. Fix `resolve_night_atomic()` key handling and verify round-to-round `night_history` preservation.
2. MM-02 is fixed in app: phase 2 now uses the save decision persisted in phase 1.
3. Remove the lobby-phase role-event drop or add an immediate `refreshPlayers()` on phase transition for all clients.
4. Repair the multiplayer backend contract:
   - add or replace `add_session_player`,
   - include `reset_players_ready` in documented setup,
   - make missing backend RPCs fail loudly instead of silently continuing.
5. Fix solo win/tie logic and no-elimination day advancement.
6. Re-enable the test target and hook `GameStoreTests.swift` into the scheme.
